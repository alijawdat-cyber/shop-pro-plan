الفصل 4 — تصميم الفلاتر الطبقية (Faceted Search)

القاعدة الذهبية (قبل الكود)

لمن تحسب عدادات نفس الخاصية (مثلاً الألوان)، لازم تشيل فلتر هاي الخاصية من الحساب وتطبق باقي الفلاتر. بحيث المستخدم يگدر يختار أحمر + أزرق + أخضر سوية.

بدون هاي القاعدة، لمن المستخدم يضغط «أحمر»، كل الألوان الثانية تختفي، وما يگدر يضيف ثاني لون.

السيناريو المستهدف — من واقع نافا

سيناريو حقيقي (nava-iq.odoo.com/shop)
المستخدم يدخل المتجر: يشوف 33 فئة + 93 ماركة + 79 لون + 32 مقاس + 114 نسيج + 63 نمط — كلها مفتوحة.

يضغط فئة «أحذية» — فوراً:
  • الماركات تتقلص من 93 لـ 8 (Nike 12 · Adidas 10 · Puma 6...)
  • الألوان تتقلص من 79 لـ 14 مع عدادات
  • النسيج يتقلص من 114 لـ 6 (جلد، قماش، كاوتشوك...)
  • المقاسات من 32 لـ 12 (بس 38-45)
يضغط «Nike» — الألوان تتقلص من 14 لـ 5. المقاسات من 12 لـ 8.

يضغط «أحمر» + «أسود» — المنتجات تفلتر لكن الألوان تبقى ظاهرة حتى يقدر يضيف/يشيل.

التصميم التقني المقترح

قبل — Odoo الحالي
t-foreach="a.value_ids"

يمرّ على كل القيم بالـ DB، بدون عدّ، بدون فلترة

بعد — Shop Pro
t-foreach="a.facet_values"

قيم مع {id, name, count, selected} من الـ controller

SQL المقترح لحساب العدادات (الأنسب أداءً) 
-- لكل خاصية نحسب منفصل مع استبعاد فلتر هاي الخاصية بالذات
SELECT
  pav.attribute_id,
  pav.id AS value_id,
  COUNT(DISTINCT ptal.product_tmpl_id) AS product_count
FROM product_attribute_value pav
JOIN product_template_attribute_line ptal ON ptal.attribute_id = pav.attribute_id
JOIN product_template_attribute_line_product_attribute_value_rel rel
  ON rel.product_template_attribute_line_id = ptal.id
  AND rel.product_attribute_value_id = pav.id
JOIN product_template pt ON pt.id = ptal.product_tmpl_id
WHERE pt.is_published = TRUE
  AND pt.active = TRUE
  AND pt.id IN (
    /* search_product.ids بعد تطبيق فلاتر أخرى غير pav.attribute_id */
  )
GROUP BY pav.attribute_id, pav.id
ORDER BY pav.attribute_id, product_count DESC;

ملاحظة: نسوي استعلام منفصل لكل خاصية مفعّلة، مع cache بالـ request.session لمدة دقيقة. على نافا (129 منتج × 7 خصائص) هذا أقل من 40ms.

مشكلة «الواجهة تتجمد» (أنت ذكرتها)

السبب الحقيقي: الضغطة = full HTTP reload = re-render كل الصفحة من السيرفر. الحل:

  1. Endpoint JSON جديد: /shop/facets يرجّع JSON بالعدادات + HTML partial للمنتجات
  2. OWL Component: ShopFacetSidebar يسمع على change ويسوي fetch
  3. Optimistic UI: لمن يضغط checkbox، فوراً نعطل الـ UI بـ disabled + skeleton بدل انتظار الاستجابة
  4. Debounce 200ms: لو ضغط 3 فلاتر بسرعة نطلب الأخير بس
  5. URL sync: history.pushState بدون reload — حتى يشير الرابط

الفصل 5 — فلتر «المتوفر بالمخزون» (الشرح العميق)

هاي نقطة دقيقة طلبت منك شرحها. Odoo ما يخزن المخزون على product.template بل على product.product (الـ Variants). فلتر «متوفر» عنده 3 مستويات من التعقيد:

بنية البيانات — لازم تفهمها

الجدولدورهحقل الكمية
product.templateالقالب (تيشيرت نايكي)qty_available = مجموع الـ variants
product.productالمتغير (تيشيرت نايكي · M · أحمر)qty_available فعلي
product.template.attribute.value (ptav)ربط variant بقيمة خاصية—
stock.quantالكمية الفعلية بالموقعquantity

3 سيناريوهات — شنو يحصل بالواجهة

السيناريو A — تيشيرت M أحمر نافد، L أحمر متوفر

قبل (بدون الفلتر)
المستخدم يشوف التيشيرت بصفحة البحث. يفتحه. يختار M + أحمر. يطلع «نفد المخزون». يزعل.
بعد (مع الفلتر + منطق Variants)
التيشيرت يظهر بالبحث (لأن L أحمر متوفر). يفتحه. يشوف M أحمر معطل/مشطوب بشاشة الاختيار (Odoo يسوي هذا أصلاً بـ excluded). L أحمر قابل للاختيار.

السيناريو B — كل الـ variants نافدة

قبل
المنتج يظهر بالمتجر. المستخدم يفتحه، كل المقاسات مشطوبة، يطلع. إحباط.
بعد (فلتر «متوفر» مفعّل)
المنتج يختفي من صفحة /shop نهائياً. ماكو إحباط.

السيناريو C — توغل «متوفر» والمنتج متوفر بلون واحد فقط

هنا قرار تصميمي مهم: المنتج عنده 5 ألوان، 4 نافدة، واحد بس (الأزرق) متوفر.
  • فلتر «متوفر» يظهره ✓
  • بس فلتر «اللون الأحمر» هل يظهره؟
    من ناحية البيانات: نعم (المنتج تبعه يحتوي أحمر بالـ attribute values).
    من ناحية المنطق: لا (الأحمر نافد فعلاً).
الحل الصح (Amazon/SHEIN): نعمل JOIN على stock.quant لفلتر قيم الخصائص بحيث بس القيم اللي عندها variant متوفر تظهر بالشريط الجانبي.

SQL المقترح للمتوفر

الـ query الأمثل لحساب قيم خاصية متوفرة فعلاً 
SELECT
  pav.attribute_id,
  pav.id AS value_id,
  COUNT(DISTINCT pt.id) AS in_stock_count
FROM product_attribute_value pav
JOIN product_template_attribute_value ptav ON ptav.product_attribute_value_id = pav.id
JOIN product_product pp ON pp.id IN (
    SELECT product_id FROM product_variant_combination
    WHERE product_template_attribute_value_id = ptav.id
  )
JOIN product_template pt ON pt.id = pp.product_tmpl_id
LEFT JOIN stock_quant sq ON sq.product_id = pp.id
  AND sq.location_id IN (SELECT id FROM stock_location WHERE usage='internal')
WHERE pt.is_published = TRUE AND pt.active = TRUE
  AND pt.id IN (/* search_product.ids مع فلاتر */)
GROUP BY pav.attribute_id, pav.id
HAVING SUM(COALESCE(sq.quantity, 0) - COALESCE(sq.reserved_quantity, 0)) > 0;

def _get_search_order(self, post):
    order = post.get('order') or 'website_sequence asc'
    # Shop Pro extensions
    sp_orders = {
        'bestsellers': 'x_sales_30d desc, website_sequence asc',
        'discount': 'x_discount_pct desc, list_price asc',
        'newest_real': 'create_date desc',
        'rating': 'rating_avg desc, website_sequence asc',
        'most_viewed': 'x_view_count desc',
        'trending': 'x_trending_score desc',
    }
    if order.split(',')[0].strip() in sp_orders:
        return sp_orders[order.split(',')[0].strip()] + ', id desc'
    return super()._get_search_order(post)
